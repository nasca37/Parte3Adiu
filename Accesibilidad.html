Accesibilidad

Informe de la accesibilidad de la página web.

Esta página web ha sido creada con la intención de realizar la práctica de Accesibilidad de la asignatura Aplicacions Distribuïdes a Internet i Interficies d'Usuari. De esta forma nos hemos centrado en crear una página sencilla, pero contiene diferentes aspectos que la mayoría de páginas web contempla como son el texto, las imágenes, gráficas, videos, tablas y formularios. Todos estos aspectos cumplen las normas de Accesibilidad del nivel AA que se declara en la normativa de WCAG 2.0.

Evaluación automática

Para cerciorarnos de este hecho hemos recurrido a una página de comprobación automática recomendada en la bibliografía de la asignatura. Esta página es conocida como achecker. A continuación se deja una tabla que para cada uno de los archivos html muestra el icono de la certificación y el enlace para descargar el pdf que corresponde a la certificación de cada página.

Obviamente como ya conocemos no es posible asegurar de forma completa que una página web asegura todas las funcionalidades necesarias para que sea lo más accesible posible para todo tipo de usuarios. Por ese motivo a continuación revisaremos algunos de los aspectos que no se pueden verificar de forma automática y requieren por tanto de un toque humano.

Evaluación Manual.

Descripciones de las imagenes.

Como podemos ver en cada uno de los pdfs que se muestran arriba vemos que un gran conjunto de los potenciales problemas tiene que ver con las descripciones de las imágenes. Estos posibles errores son que la descripción sea muy corta, que sea muy larga, que no se adecúe al contenido de la imagen o que directamente no sea necesario por X motivos. Como la página ha sido creada por nostros al dedo, sabemos que el contenido de cada imagen es el adecuado y tiene un tamaño que no se pasa, pero que es suficiente para explicar cada imagen.

Listas visuales.

Otro posible error que se puede encontrar en un aplicativo web es que las listas que se forman visualmente no puedan ser analizadas por un text to speach engine. En nuestro caso cada una de las listas que aparecen dentro de la web viene correctamente enlazas entre ellas haciendo uso de las herramientas de HTML5 como son las etiquetas ul y li.

Headers

Como podemos observar en los posibles problemas que puede contener la página web, uno de ellos consiste en la ocurrencia de que el header no esté bien encapsulado creando así una situación donde la página no muestre como es debido. Como se puede observar tanto en el código como de forma visual el header está bien encapsulado y muestra correctamente. El header es compartido por todas las planas web, conteniendo los 4 links y el nombre de la web.

Texto en Imagenes

Hay imágenes en la web que tienen texto dibujado en ellas y que por tanto pueden suponer un problema para las personas con problemas de visibilidad. Esto implica que las personas con este tipo de discapacidades no puedan diferenciar de forma correcta la imagen. Para solucionar este problema hacemos uso del atributo alt de las imágenes de forma que se describa lo mejor posible la imagen. Además las pocas imágenes que tienen estas características están seleccionadas para que el poco texto que contengan sea bien grande y con buen contraste.

Scripts

A veces los scripts de una página pueden modificar el html de esta y crear problemas para diferentes usuarios. En nuestro caso como la aplicación es en sí una estructura bien formada sin el uso de Javascript no nos encontramos con este error.

Nombre de los enlaces

Los nombres de los enlaces y los textos anclados a estos definen y se relacionan correctamente respecto a la plana a la que se envía al usuario. Ejemplos como el texto Ranking y su enlace ranking.html o Sugerencias y sugerencias.html muestran que el texto identifica al link.

Texto legible

Nos hemos asegurado que todo el texto que aparezca en la página sea de un color afilado como el negro de forma que en todo momento el usuario sea capaz de leer el texto. Además el tamaño de las letras siempre ha sido diseñado tirando hacia lo más grande posible de forma que personas con dificultades para leer lo tengan un poco más fácil.

Contraste

El contraste puede ser uno de los errores más típicos en el desarrollo de aplicaciones. En nuestro caso el analizar de contraste del achecker no ha saltado mencionando ningún error y además la extensión Funklify que permite analizar el contraste y no ha saltado en ninguna de nuestras páginas. Cabe mencionar que si un problema pudiera surgir seria con el recuadro de color gris que se muestra junto al enlace de los artículos en el índice. Este recuadro nos hemos asegurado que sea lo más claro posible de forma que el contraste se mantiene, pero se puede enseñar la estructura que sigue el elemento del body del índice.

Elementos comunes

Todos los elementos que se repiten en las diferentes planas aparecen en el mismo orden. Siendo los elementos repetidos el header y el footer. Estos elementos se repiten siempre igual en todas las planas, siendo el orden: Header, body y footer. Esto tiene la intención de mejorar la navegabilidad de forma que ayuda a entender la estructura de la aplicación.

Pruebas de usuario

En el mundo del desarrollo de aplicaciones es siempre necesario un paso que permita asegurarse que el trabajo que se ha realizado para la aplicación ha sido satisfactorio y es correcto para los usuarios objetivos de la aplicación. De igual forma cuando trabajamos para asegurar que las mediadas de accesibilidad se cumplen uno de los pasos en la evaluación es un conjunto de pruebas de usuario donde el aplicativo web es usado por personas con diferentes discapacidades.

En nuestro caso donde nosotros no formamos una compañía que tenga la capacidad de organizar unas pruebas de usuario a una escala media o grande, hemos decidido usar un simulador de discapacidades en forma de plug-in de navegador conocido como Funkify. Esta herramienta ofrece una gran variedad de ajustes ya creados para simular discapacidades y además permite crear configuraciones mixtas. Hemos hecho uso de esta extensión para simular las siguientes discapacidades:

  • Daltonismo. (Visión limitada a la escala de grises).
  • Discapacidades visuales de desenfoque (La pantalla se ve borrosa con un factor del 60%).
  • Parkinson.

Estas pruebas han consistido en usar la página web para llegar al formulario de enviar y rellenarlo de forma correcta. Como la página web en sí es un esqueleto, es decir, no tiene un servidor que se encargue de gestionar las peticiones y de realizar o guardar las funcionalidades, no podemos ver como se actualizan los artículos. Aun con esto la intención ha sido comprobar que los campos han podido ser rellenados de forma correcta. Estas pruebas las hemos realizado los dos miembros del grupo por separado para asegurar que no hubiera influencia del uno al otro y dos personas externas que nos han hecho el favor de realizar la prueba. Estas personas han uso el mismo simulador con la misma configuración.

Los resultados han sido más o menos los que esperábamos. Los cuatro usuarios han completado la tarea de forma satisfactoria. Aun así somos conscientes que este es debido a la simplicidad de la página y al hecho de que no es lo mismo hacer pruebas con un grupo representativo de personas con discapacidades varias que usar un grupo de 4 personas que usan un simulador de navegador.

Conclusiones

Creemos que a nivel personal la práctica ha sido un éxito. Se ha conseguido el nivel AA en todas las páginas de la aplicación y el AAA en la mayoría. Sabemos que esto es debido a que en general la página es en sí un esqueleto que carece de uso real, pero aun así creemos que se muestran de forma representativa la mayoría de elementos que se pueden observar en aplicaciones reales como pueden ser las tablas, imágenes, gráficos, videos y formularios.
